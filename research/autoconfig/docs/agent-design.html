WinML CLI Agent Design

Concise architecture for the agent layer on top of winml-cli, with autoconfig as a targeted sub-tool.

Draft2026-06-21Research POC

1. Positioning: Why an Agent Layer

Olive vs winml-cli

Olive is pass-centric and expert-driven. winml-cli is developer-centric and workflow-driven (config → build → validate), with Windows hardware diversity as a first-class concern.

Agent value

The agent focuses on diagnosis, explanation, and decision guidance. Compute-heavy search stays in deterministic tooling (autoconfig loop), then gets interpreted for developers.

3. Agent Capability Model (5 Types)

1) Diagnostic Agent (P0)

Explain failure root cause from build/eval/analyze/profiler signals and give executable fixes.

2) Decision Guidance Agent

Run small comparative probes and present tradeoffs (latency, accuracy, portability) as actionable choices.

3) Cross-Device Confidence Agent

Estimate deployment confidence across heterogeneous Windows devices and produce fallback chains.

4) Regression Detection Agent

Track perf/accuracy drift across runtime, driver, and dependency changes with statistically valid comparisons.

5) Model Recommendation Agent

Map product intent + hardware profile to a practical model candidate and a ready-to-run command path.

4. Autoconfig's Role

Autoconfig is a sub-tool, not the primary UX entry. The agent invokes it only when focused search is needed, then returns a human-readable report with rationale and alternatives.

  • Targeted sweep over EP × opset × graph options (not blind exhaustive search).
  • Mandatory correctness gate via winml eval.
  • Report includes a feasible-options comparison table for decision-making.

5. Integrated Core Ideas from Related Designs

Cross-device core (from cross-device-design)
  • Single-device champion configs do not answer fleet deployment questions.
  • winml serve Phase 0 can act as distributed workers (no new tune protocol required).
  • Need a joint objective over heterogeneous devices, not independent local optima.
  • Add a device axis to KB confidence and portability claims.
Self-evolution core (from self-evolution-design)
  • Use paired A/B protocol to reduce thermal/sequence bias.
  • Adaptive sampling replaces fixed session counts.
  • Confidence ladder L1→L5 governs promotion; KB should hold stable evidence tiers.
  • Automated finding promotion closes the loop and shortens future sweeps.

7. Open Questions

  • Surface shape: standalone winml agent vs agent-assisted existing commands.
  • Cross-device execution in practice: lab fleet, cloud runners, or hybrid.
  • Offline mode: deterministic, non-LLM fallback for constrained environments.
