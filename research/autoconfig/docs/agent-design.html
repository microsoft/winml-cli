WinML CLI Agent Design Doc¶

Status: Draft (research POC) · Updated: 2026-06-21

Problem statement¶

winml-cli already provides strong primitives (inspect, analyze, build, eval, perf), but most developers still cannot reliably answer expert-level questions: why a model fails on a target EP, which configuration is safe to ship, and whether a local optimum generalizes across Windows hardware diversity.

Current optimization practice is often single-device and manually interpreted, which creates three recurring issues:

  • Decision fragmentation: command outputs are available, but reasoning is not systematized.
  • Portability blind spots: a config that wins on one device can regress on another.
  • Confidence gaps: noisy measurements and insufficient promotion gates can produce unstable recommendations.

Design principles¶

  1. Agent for judgment, tools for computation: keep heavy search deterministic and use the agent for diagnosis, prioritization, and explanation.
  2. Lifecycle orchestration first: one orchestrator role spans Intake → Insight → Opt Loop → Outcome.
  3. Evidence over intuition: all recommendations are backed by validation signals and confidence semantics.
  4. Cross-device by default: design for deployment fleets, not only the developer machine.
  5. Self-evolving knowledge: findings are promoted through confidence levels before broad reuse.
Lifecycle visual reference: autoconfig_diagram.html.

Solution¶

Introduce an agent layer on top of winml-cli commands. The layer provides five capability types and invokes autoconfig as a targeted sub-tool when controlled search is needed.

Agent capability model¶

Agent typePrimary roleTypical trigger
Diagnostic Agent (P0)Explain root cause and propose executable fixesBuild/eval/analyze failure
Decision Guidance AgentCompare feasible options with explicit tradeoffs"Which config should I pick?"
Cross-Device Confidence AgentEstimate portability and propose fallback chains"Will this ship safely across devices?"
Regression Detection AgentDetect and attribute perf/accuracy driftRuntime/driver/toolchain changes
Model Recommendation AgentMap use-case + hardware profile to model path"Which model should I start with?"

Autoconfig positioning¶

Autoconfig is a sub-tool, not the primary UX entry. The agent uses it for targeted sweep over EP × opset × graph options, then returns an explainable report with a feasible-options comparison table. Correctness validation (winml eval) is mandatory before recommendation.

How it works¶

Lifecycle orchestration¶

The orchestrator controls phase transitions and loop gates. Explorer/Optimizer/Reviewer execute the optimization loop, while outcome synthesis consolidates recommendation and evidence into final outputs.

Cross-device integration¶

From cross-device-design.html: treat winml serve Phase 0 endpoints as distributed workers, optimize with joint objective across devices, and add a device axis to portability confidence.

Self-evolution integration¶

From self-evolution-design.html: use paired A/B protocol and adaptive sampling to stabilize conclusions, then promote findings through L1→L5 confidence levels before broad KB reuse.
