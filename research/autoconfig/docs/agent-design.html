WinML CLI Agent Design

Concise architecture for the agent layer on top of winml-cli, with autoconfig as a targeted sub-tool.

Draft2026-06-21Research POC

1. Problem statement

Developers can run winml-cli commands, but they still lack expert judgment: why a model fails on a specific EP, which config is actually safe to ship, and how a local optimization behaves on other Windows devices.

Current pain
  • Decision logic is manual and fragmented across inspect/analyze/build/eval/perf.
  • Single-device optimization cannot answer fleet-level deployment confidence.
  • Perf conclusions are noisy without confidence gating and robust bench protocol.
Why this matters
  • winml-cli already has strong primitives, but no unified reasoning layer.
  • Wrong config choices create hidden regressions and weak portability.
  • Without explainable output, developers cannot trust or audit recommendations.

2. Design principles

  • Agent for judgment, tool for computation: keep heavy search deterministic; use the agent for diagnosis, prioritization, and explanation.
  • Orchestrator as lifecycle spine: one control role across Intake → Insight → Opt Loop → Outcome (see autoconfig_diagram.html).
  • Evidence-first decisions: every recommendation carries validation evidence, not point estimates.
  • Cross-device by design: optimize for deployment fleet behavior, not only the dev machine.
  • Self-evolving knowledge: findings move through confidence levels before becoming reusable KB guidance.

3. Solution

Introduce a concise agent layer with five capability types, with autoconfig as a targeted sub-tool.

1) Diagnostic agent (P0)

Explains failures and proposes executable fixes from build/eval/analyze/profiler evidence.

2) Decision guidance agent

Runs focused comparisons and returns explicit tradeoff guidance.

3) Cross-device confidence agent

Estimates portability and generates fallback chains across heterogeneous devices.

4) Regression detection agent

Detects drift after runtime/driver/toolchain changes with statistically valid checks.

5) Model recommendation agent

Maps product intent + hardware profile to practical model choices and runnable paths.

Autoconfig positioning: sub-tool, not primary entry point. It runs targeted EP/opset/graph-option search and returns a report with a feasible-options comparison table.

6. How it works

6.1 Lifecycle orchestration

The Orchestrator controls all phases and hands off work to Explorer/Optimizer/Reviewer inside the optimization loop, then merges outcomes into a final recommendation.

6.2 Cross-device integration (from cross-device-design)

  • Treat winml serve Phase 0 endpoints as distributed workers.
  • Optimize against a joint multi-device objective, not independent local optima.
  • Track portability with a device axis in KB confidence semantics.

6.3 Self-evolution integration (from self-evolution-design)

  • Use paired A/B protocol to reduce thermal and sequence bias.
  • Use adaptive sampling when variance is high.
  • Promote findings through L1→L5 confidence gates before broad reuse.
