# ModelKit CLI - Command Quality Checklist

A concrete, testable set of rules each command runs through before being declared "Ready". One pass per command. Rules are written as positive assertions answerable YES / NO.

The 6 sections below are the single source of truth and are written for human review. The agent workflow that applies these rules systematically lives in [quality-check-skill.md](quality-check-skill.md). The latest audit produced by that workflow is [CLI_quality_check_report.md](CLI_quality_check_report.md).

---
## 1. Discoverability

A user shall be able to figure out how to run any command from the tool itself — `--help`, `--version`, error messages, and onboarding commands — without reading the README, browsing the source code, or asking a teammate. Help is example-led, written in user language, and never advertises surface that does not work.

| # | Rule | Examples |
|---|------|----------|
| 1.1 | **`--help` is example-led, written in user language, and complete.** Each command's `--help` shall open with a one-line purpose statement in user language and at least one runnable, copy-pasteable example. It shall describe user-visible outcomes — not internal class names, private symbols, internal pipeline steps, or third-party project classes. Top-level `winml --help` shall additionally show a starter hint (first command to run), group subcommands by purpose when there are many, and end with a `LEARN MORE` footer (docs URL + bug-tracker URL); `--version` shall include build/runtime provenance (Python, OS, ORT, key EP versions, install path). | Reference: `ollama run --help` (example-first), `gh --help` (`CORE COMMANDS` + `LEARN MORE`), `gh --version` (multi-line provenance). Forbidden in help text: `WinMLSession`, `_run_onnx_benchmark`, `TasksManager`, `(via auto_class)`, internal step listings ("1. Model Preparation, 2. Input Generation, …"). |
| 1.2 | **Every flag visible in `--help` is fully implemented.** No flag shall print "not yet supported", "TODO", or "Falling back to default behavior" at runtime. In-progress flags are marked `hidden=True` until they work; once shown, the user's intent is honored. | If `--torch-module` is shown in help, passing `--torch-module my.module` actually uses that module rather than silently falling back to the default export. |
| 1.3 | **Structured-value flags ship with a complete schema or working example.** When an option accepts JSON, `key=value` lists, role mappings, or any non-trivial value, `--help` shall list every accepted key (or show a complete, shell-safe example) so the user can fill it in without reading source code. | `--shape-config` shows a path-based example (`--shape-config shape.json`) plus a link to the schema, not an unquoted-brace JSON snippet that PowerShell misparses. For `winml eval -m`, help enumerates the accepted role keys (`text_encoder`, `text_decoder`, `vision_encoder`, …) and shows a composite-model invocation. |
| 1.4 | **Naming follows real-world CLI conventions and does not collide with adjacent tools.** Subcommand and flag names shall match what a user coming from `git`, `gh`, `kubectl`, `cargo`, Olive, or HuggingFace expects. Short flags follow the project-wide reserved table; British/American spelling is uniform; type-name placeholders (`TEXT`, `PATH`, `INTEGER`, `TASK`) are consistent. | `winml config` configures the CLI itself (à la `git config`, `gh config`, `aws configure`); per-model build JSON generation lives under a different verb. `quantize` is spelled the same way in every command (no `quantise` in one, `quantize` in another). `-t` always means `--task`. |
| 1.5 | **Typos get a "did you mean…?" suggestion, not a dead end.** An unknown subcommand, an unknown flag, or an invalid value for a `Choice` option shall produce a one-line suggestion when a close match exists, instead of a bare "no such command/option" error. | Reference: `git`, `gh`, `cargo`, `kubectl`. `winml exprt` suggests `winml export`. `winml export --shape` suggests `--shape-config`. `--device gpyu` suggests `--device gpu`. |
| 1.6 | **Environment variables are documented in help.** Every env var the tool reads (`HF_HOME`, `HF_HUB_OFFLINE`, `HF_TOKEN`, `WINML_*`, proxy vars) shall be listed in top-level `--help` (or a per-command "Environment" section). | Reference: llama.cpp documents every `LLAMA_*` env var inline; `gh` documents `GH_*`. A user who sets `WINML_DEVICE=npu` in their shell sees that mentioned in `winml --help`. |

## 2. Consistency

A user who has learned one `winml` command should be able to predict how every other command behaves. Flags, names, defaults, and verbosity work the same way across the entire tool. Sibling commands never disagree on the meaning of the same word.

| # | Rule | Examples |
|---|------|----------------------------------------|
| 2.1 | **Same flag = same meaning.** A flag with the same long name shall have the same purpose, the same default, the same accepted values, the same short form, and the same help wording in every command that exposes it. | `--model` / `-m`, `--device` / `-d`, `--ep`, `--task` / `-t`, `--output` / `-o`, `--output-dir`, `--config` / `-c`, `--precision` / `-p`, `--format`, `--verbose` / `-v`, `--quiet` / `-q` |
| 2.2 | **One canonical name per concept.** Each concept shall be expressed by exactly one flag spelling across the tool. Older spellings are kept only as deprecated, hidden aliases on a published removal schedule. | Quantization toggle → `--quantize` / `--no-quantize` (not `--no-quant`, not single-direction `--no-quantize`). Single-file output → `--output`; directory-tree output → `--output-dir`. |
| 2.3 | **Same concept, same default.** A shared flag shall have the same default value in every command. If a specific command must deviate, the deviation is called out explicitly in that command's `--help`. | `--device` defaults to `auto` everywhere. Explicit `--device npu / gpu / cpu` warns when unavailable but never silently falls back. `--quiet` defaults to off everywhere. |
| 2.4 | **Verbosity is uniform.** `--verbose` / `-v` shall be stackable (`-v`, `-vv`, `-vvv`) in every command and shall control the same thing (general logs → debug logs → tracebacks). `--quiet` / `-q` shall be a boolean flag in every command and shall suppress all non-error output. | A user typing `winml <any-command> -vv` sees the same depth of detail across commands. Default verbosity is identical across commands. |
| 2.5 | **Reserved short flags.** Single-letter flags `-c`, `-d`, `-m`, `-o`, `-p`, `-q`, `-t`, `-v` shall map to their canonical long names. A command must not bind a reserved short flag to a different long name. | `-t` always means `--task`; it must not mean `--text` or `--model-type` in another command. `-p` always means `--precision`; it must not mean `--preset`. Existing exceptions are grandfathered and tracked for migration — they are not a license for new violations. |
| 2.6 | **Help text reads the same.** The description for a shared option shall be identical across every command's `--help`. Per-command context may be appended (e.g. "For `quantize`, this controls the calibration set"); it must not replace the canonical wording. | Reading `--device --help` in `compile` and in `eval` produces the same first sentence. |
| 2.7 | **Outputs reflect user input *and* actual execution, in one canonical spelling.** Every output channel — console logs, result tables, JSON, on-disk config and report files — shall report the device, EP, and precision *actually used*, using the same canonical spelling that the CLI accepts. When a requested value was overridden (e.g. NPU → CPU fallback), both the requested and effective value are recorded and the override is surfaced at runtime. | `--ep qnn` is reported as `qnn` in every log line, JSON field, and report file (no mix of `qnn` / `QNNExecutionProvider` / `QNN`). Running `--device npu` on a non-NPU machine produces output where `requested_device=npu` and `effective_device=cpu`, plus a runtime warning. |
| 2.8 | **One output format across the tool.** Log timestamp / level prefix / color scheme, table column conventions, JSON schema for shared fields (`command`, `version`, `model`, `device`, `ep`, `precision`, `started_at`, `duration_s`, `exit_code`), and stream routing (data → stdout, logs/progress/errors → stderr) are identical across commands. | `winml eval --format json` and `winml perf --format json` share field names and value spellings. A "device" column is spelled `device` in every table. A failed measurement renders as `FAIL` everywhere. |

## 3. UX

The tool provides a graceful interaction experience: errors guide the user toward a fix instead of dead-ending, output volume is bounded and trustworthy at default verbosity, results render cleanly across terminals and pipes, and the tool stays out of the user's working directory unless asked.

| # | Rule | Examples |
|---|------|----------|
| 3.1 | **Errors are user-friendly and actionable.** Every error shall name *what failed*, *why*, and *what to try next* ("Run X" / "Install Y" / "Try `--device cpu`"). No raw Python traceback at default verbosity — tracebacks are gated behind `-vv` / `--debug`. The diagnosis shall be correct: a missing local file is reported as file-not-found, a bogus HF id as model-not-found, an unknown task with the valid set listed — not all three as "Network error" or "Unexpected error". A "did you mean…?" suggestion is offered for typo'd subcommands, flags, or `Choice` values. Required-but-missing options produce a per-command example, not a bare `Missing option '--model'`. | Reference: Ollama (`could not connect to ollama server, run 'ollama serve' to start it`). Forbidden: `Network error: ./does-not-exist.onnx is not a local folder. If this is a private repository, …` for a missing local path. Acceptable: `Error: --model/-m is required.\n  Example: winml export -m microsoft/resnet-50 -o resnet50.onnx`. |
| 3.2 | **Default-verbosity log volume is bounded.** A successful invocation shall not flood the console with internal-library WARNINGs, ISO timestamps, duplicate messages, or jargon (`PatternMatcher`, `IHV`, `_run_onnx_benchmark`). Library-internal warnings are demoted to DEBUG; surfaced only at `-v` / `-vv`. Warnings emitted multiple times for the same condition are de-duplicated. The user shall be able to scroll a typical command's output in one screenful and trust that anything still printed at default verbosity is something they need to see. | A typical `winml perf` run prints a banner, a labeled progress region per stage, and a result table — not 4–7 unrelated WARNINGs interleaved with the result. `--quiet` is honored uniformly across commands. |
| 3.3 | **Result rendering is human-readable, scriptable, and well-behaved.** Result tables, banners, and emojis shall use plain English (no `TasksManager`, `(via auto_class)`, `WinMLSession`); truncated values show `…` or "+N more"; data goes to `stdout` while logs / progress / banners / errors go to `stderr` so the command is pipeable (`<cmd> | jq .`, `<cmd> > out.json`). Output adapts to its destination: `NO_COLOR=1`, `CI=true`, `TERM=dumb`, `--no-color`, and `not isatty(stdout)` disable color and emoji; on Windows, the command falls back to ASCII box-drawing when the console codepage is not UTF-8 (no `UnicodeEncodeError` crash, no mojibake, no literal `\u2550…\U0001f4cb`). | Reference: `clig.dev` (data → stdout, everything else → stderr), `kubectl` / `gh` `NO_COLOR` handling. A piped run produces parseable JSON without ANSI escapes; a default Windows PowerShell run renders without crashing. |
| 3.4 | **The tool does not pollute the user's working directory or terminal.** No command shall write a default output file into the current working directory. Output goes to an explicit `-o` / `--output` path, or — when auto-named output is allowed — under `~/.cache/winml/` or `./.winml/runs/<timestamp>/`. A pre-existing output file is not overwritten silently; `--force` / `-f` is required, or the command errors with a clear message. | `winml perf` without `-o` shall not drop a `{slug}_perf.json` next to the user's `pyproject.toml`. `winml export -o model.onnx` against an existing file errors with "File exists; use --force to overwrite", not silent overwrite. |
| 3.5 | **Security-sensitive flags warn before doing work.** Any flag that loosens a default safety boundary (e.g. `--trust-remote-code`, `--insecure`, anything that allows execution of code shipped with a model) shall emit a one-line stderr warning naming what is being permitted, before the risky work begins. The warning is not suppressed by `--quiet`. | `winml export -m <repo> --trust-remote-code` prints `WARNING: --trust-remote-code allows running custom Python from <repo>. Proceed only if you trust the source.` before downloading. |

## 4. Reliability & Performance

The command behaves like an API: bad input is rejected up front with a clear message, the user (or a CI script) can trust that "exit 0" means the command actually succeeded, and the tool is fast where it should be fast and gives feedback where it cannot.

| # | Rule | Examples |
|---|------|----------|
| 4.1 | **User inputs are validated at command entry.** Every option shall be validated before any model download, optimization, or compilation work begins. Bad values (unknown task, missing file, non-existent HF id, unwritable output path) fail fast with a single-line error. The command never writes a "successful" artifact that the rest of the pipeline cannot consume, and never exits 0 after a silent fallback. | `winml perf --device gpu` against a machine with no GPU shall fail in the first second, not after 33 seconds of progress bar. `winml export -o /read-only/path/model.onnx` errors at entry via a touch-and-delete writability probe, not at the serialization step after the work is done. |
| 4.2 | **Conflicting or unsupported flag combinations are rejected, not silently overridden.** When two flags conflict (e.g. `--device cpu --ep qnn`), or a requested combination is not supported on the host (EP not installed, hardware not present), the command shall reject the input with `exit code 2` and a message naming both flags and a suggested resolution. No flag silently overrides another; no warning-then-continue. | Forbidden: `WARNING: Device 'npu' requested but no compatible EP found.` followed by `Config saved to: cfg.json` and exit 0. Acceptable: `Error: --device cpu and --ep qnn are incompatible. QNN runs on NPU; pick one of --device npu, --ep CPUExecutionProvider.` |
| 4.3 | **Quick commands return quickly.** `winml --help`, `winml --version`, and any command that only inspects local state (e.g. `winml sys`, dry-run modes) shall return in **under 500 ms warm**. Heavy dependencies (`torch`, `onnxruntime`, `transformers`, `optimum`) are imported lazily inside command bodies, never at module top level. A CI gate enforces the budget. | Reference: `gh --help` ~50 ms, `uv --help` ~20 ms, `ollama --version` ~100 ms. `winml sys` reads package versions via `importlib.metadata.version(...)`, not by importing the packages. |
| 4.4 | **First user-visible output appears within 500 ms of launch.** A long-running command shall print a banner, status line, or spinner within that window — never a blinking cursor. Every multi-second stage shall show labeled progress (named unit of work, percentage or count, ETA when available). Progress regions are rendered through one shared progress mechanism — no mid-flow switch from Rich to monochrome ASCII bars, no double-printed `100%` lines, no unlabeled iteration counters. A `winml inspect` invocation never sits silent for 21 of 23 seconds; an `Optimizing graph (122 passes)` bar is labeled, not a bare `122/122`. | Reference: Ollama's per-layer progress (named layer + bar + ETA). |
| 4.5 | **Exit codes follow a documented contract.** Every command shall exit with a code from the project-wide table; bare `sys.exit(N)` is forbidden. CI scripts shall be able to distinguish success, negative result, usage error, hardware/EP unavailable, network error, I/O error, and internal error from the exit code alone. | Project-wide table: `0` success; `1` operation completed but result is negative (model unsupported, regression); `2` usage / validation error; `3` hardware / EP unavailable; `4` network / download error; `5` I/O error; `70+` internal error. Errors are raised via `click.ClickException` or a `ModelKitError` subclass, not bare `sys.exit()`. |

## 5. Functional Correctness

The command does what its help and docs say it does, on every supported model format and every supported execution provider. Same input + same environment produces the same result on repeated runs. An interrupted command never leaves the system in a wrong state.

| # | Rule | Examples |
|---|------|----------|
| 5.1 | **The command handles every supported model format.** Where applicable, the command shall correctly process non-quantized models, multiple opset versions in the supported range, QDQ-quantized models, and compiled (EP-context) models. Format detection is automatic; format-specific behaviour is exercised by the test suite, not just the most common path. | A command that takes an `.onnx` input shall succeed on a vanilla FP32 export, a QDQ INT8 quantization, an opset-17 model, an opset-21 model, and an EP-context-compiled model — without the user having to pass a `--format` hint. |
| 5.2 | **The command works across every supported Execution Provider.** Where applicable, the command shall produce correct results on every EP it advertises (CPU, DML, QNN, OpenVINO, VitisAI, NvTensorRTRTX, …). EP-specific behaviour is keyed off a registry, not hard-coded by string in the command file; adding a new EP does not require editing the command. | `winml perf --ep <X>` produces a valid result for every `<X>` listed in `--ep`'s help. No `if ep == "VitisAIExecutionProvider": …` branches scattered through command code. |
| 5.3 | **Cached artifacts are keyed correctly and invalidated correctly.** When a command writes to or reads from a cache, the cache key shall include every input that affects the output (model id + revision, device, EP, precision, opset, relevant flags, tool version). A change in any keyed input shall cause a cache miss and a fresh compute; an unchanged input shall cause a cache hit. | Re-running `winml build` with the same flags hits the cache; changing `--precision` from `int8` to `fp16` misses the cache and rebuilds. A tool-version bump invalidates stale entries; the cache never returns an artifact built for a different EP. |
| 5.4 | **Cache state is visible and manageable.** Cache location, hit/miss status, and entry size shall be observable by the user. A `winml cache <list\|info\|clean\|prune>` subcommand (or equivalent) shall let the user inspect what is cached, total disk use, and clear entries safely. Cache hits and misses are surfaced at `-v`. | `winml cache info` prints the cache root and total size; `winml cache list` shows entries with their keys; `winml cache clean --older-than 30d` prunes safely. `winml build -v` logs `Cache hit: <key>` or `Cache miss: building from scratch`. |
| 5.5 | **Idempotency: same input + same environment produces the same result.** Repeated runs shall be safe and consistent. Pre-existing output files are not overwritten silently; `--force` / `-f` is required, or the command errors with a clear message. Where stochasticity is unavoidable (calibration sampling, seed-dependent operations), a `--seed` flag is exposed and recorded in the output manifest. | Running `winml quantize -m model.onnx -o q.onnx` twice in a row produces the same `q.onnx` (or errors on the second run without `--force`). `winml eval --seed 42` reproduces the same metrics across machines. |
| 5.6 | **Interrupt safety: `Ctrl+C` leaves a clean filesystem.** A SIGINT (or any unhandled crash) mid-run shall not leave partial output files, half-written caches that a subsequent `--use-cache` would load as valid, or residual temp files in the user's CWD or output directory. Output paths use atomic-rename semantics (`tempfile` + `os.replace`); a SIGINT handler removes in-progress artifacts and exits 130. | `winml export` interrupted halfway leaves no `.onnx` / `.onnx.data` partial. `winml build` interrupted leaves no half-written cache entry that the next `--use-cache` run treats as a hit. |

## 6. Install & Environment

The command works on a fresh `pip install` without manual setup, behaves predictably with respect to environment variables, and does not bloat the wheel.

| # | Rule | Examples |
|---|------|----------|
| 6.1 | **Every required asset is available on a fresh install.** Rule files, opset zips, default configs, and any other data the command depends on shall ship inside the wheel, *or* be lazy-downloaded on first use with a single-line progress bar and cached under `~/.cache/winml/`. The command shall not instruct the user to run a `scripts/*.py` file that does not exist in a `pip install` checkout. | A user `pip install`s the wheel and runs `winml perf` without prior setup; required rule zips are either bundled or fetched on first use. No `Run 'uv run python scripts/download_rules.py' to download rule files` message. |
| 6.2 | **Optional dependencies are detected at command entry with an actionable message.** If an optional package (`onnxruntime-qnn`, `optimum`, `openvino`, …) is required by the requested operation and missing, the command shall detect this at entry and print a one-line `<cmd> requires <pkg>. Install with: uv pip install <pkg>` message, exiting with the documented "hardware / EP unavailable" code — never a raw `ImportError` traceback mid-run. | Running an EP-specific command without that EP's runtime installed produces `winml perf requires onnxruntime-qnn. Install with: uv pip install onnxruntime-qnn` and exits 3, not a 30-line traceback ending in `ModuleNotFoundError: No module named 'onnxruntime_qnn'`. |
| 6.3 | **Environment variables are honored consistently.** The command shall honor every documented env var (`HF_HOME`, `HF_HUB_OFFLINE`, `HF_TOKEN`, `WINML_*`, proxy vars). An air-gapped / proxied user shall be able to opt into offline mode without code changes, either via an explicit flag or via `HF_HUB_OFFLINE=1`. | `HF_HUB_OFFLINE=1 winml inspect -m <local-id>` succeeds against the local cache without contacting the network. `HTTP_PROXY` is respected for HF downloads. |
| 6.4 | **Wheel size stays within the documented budget.** When data files (rule zips, op tables, default configs) ship inside the wheel, total package-size growth shall be tracked per release and stay under the project's documented size budget. Newly-bundled assets that would push the wheel over budget shall be moved to lazy-download instead. | A new opset's rule zip is bundled only if the wheel-size CI gate still passes; otherwise it is fetched on first use and cached under `~/.cache/winml/`. |
