# src/winml/modelkit/sysinfo/device.py

## TL;DR
File deleted in its entirety (191 lines). Its contents have been split three ways and redistributed to *natural homes*: hardware-detection logic moves up to `sysinfo/hardware.py`, EP-discovery logic moves out to `session/ep_registry.py`, and the device-routing entrypoint (`resolve_device`) is reborn in `session/ep_device.py` as a typed `(ep, device) -> EPDevice` resolver (with the string-only auto-pick path extracted to a sibling `auto_detect_device()` helper). The hardcoded `_EP_DEVICE_MAP` dict is dissolved — its data is now derived from the `EPDeviceSpec` catalog in `session/ep_device.py`.

- Post-state symbols: none (file gone)

## Role before vs after
- Before: A 191-line module sitting *inside* `sysinfo` but doing EP-vs-device routing. It owned a hardcoded `_EP_DEVICE_MAP` (NVIDIA/AMD/Qualcomm/MS/Intel + CPU), a derived inverse map, a private hardware-presence detector, a cached EP discovery (combining `WinMLEPRegistry` + `ort.get_available_providers()`), and the public `resolve_device("auto"|...)` that returned `(chosen_device, available_list)` after cross-checking EP availability. Architecturally this conflated three concerns: (1) static EP↔device knowledge, (2) live hardware introspection, (3) live EP-plugin introspection.
- After: The three concerns are separated:
  1. Static EP↔device knowledge -> `EPDeviceSpec` catalog (an ordered tuple of dataclasses) in `session/ep_device.py`. The dict-shape API (`get_ep_device_map`) is *not* preserved; consumers use catalog helpers (`default_device_for_ep`, `eps_for_device`, etc.).
  2. Live hardware introspection -> `sysinfo/hardware.py::get_available_devices` (now public, was private `_get_available_devices`).
  3. Live EP-plugin introspection -> `session/ep_registry.py::available_eps()` (now public, was private `_get_available_eps`; the `functools.lru_cache(maxsize=1)` semantics were carried over).
  Plus: the public entrypoint is reshaped. The bare name `resolve_device` is reused in `session/ep_device.py` for a new typed resolver `resolve_device(ep=None, device=None) -> EPDevice`; the string-only auto-pick behaviour of the old `sysinfo.resolve_device("auto")` lives in a sibling helper `session.auto_detect_device() -> str` (also reachable from `resolve_device(device="auto")` via internal delegation).

## Behavior / contract changes
- **The bare name `resolve_device` is reused for a fundamentally different function.** Before: `sysinfo.resolve_device(device: str = "auto") -> tuple[str, list[str]]` — pure device-string resolution, no DLL loads, only `ValueError` on failure. After: `session.resolve_device(ep: str|None = None, device: str|None = None) -> EPDevice` — a typed resolver that performs a deduction matrix over `(ep, device)` pairs, **registers an EP plugin** via `WinMLEPRegistry` as a side effect, and raises one of five exception types (`EPNotDiscovered`, `EPRegistrationFailed`, `DeviceNotFound`, `AmbiguousMatch`, plus `ValueError`). The string-only auto-pick contract of the old function lives in `session.auto_detect_device() -> str`. Callers migrating from the old function must pick the right replacement based on whether they want an `EPDevice` (use `session.resolve_device`) or just a category string (use `session.auto_detect_device`).
- **OpenVINO multi-device handling changed.** Old code stored `"npu/gpu/cpu"` as a single string value for OpenVINO and excluded it from the inverse map via `if "/" not in _device`. The new catalog encodes OpenVINO as three separate `EPDeviceSpec` rows (one per category), so OpenVINO *is* a candidate EP for all three device categories in the new resolution path. For auto-resolution on a host with only OpenVINO installed, the resolved device may differ between old and new code.
- **Hardcoded EP list expanded.** Old `_EP_DEVICE_MAP` had 8 entries; new catalog has 13 (notably adds `CUDAExecutionProvider`, `TensorrtExecutionProvider` — see commit body, "cuda/tensorrt added to `_SHORT_TO_FULL`"). `NvTensorRtRtx` casing bug fix mentioned in the commit message lives in the catalog (verified via `ort.get_all_providers()`).
- **No `get_ep_device_map` replacement exposed.** Tools/tests that called this public function for inspection must now traverse the catalog. The commit body explicitly directs: "do not import private symbols (`_EP_TO_DEVICE`, `_DEVICE_TO_PROVIDER`, `_SHORT_TO_FULL`) outside session/ep_device.py — use the session/ facade and public helpers."
- **Module-level FIXME removed.** The long comment in the old file (lines 12–35) explaining *why* `_EP_DEVICE_MAP` had to be hardcoded (PyPI ORT lacks `get_ep_devices()` until Windows ML build lands on Win 11 25H2+) is **lost** — the same constraint applies to the new catalog but the rationale is no longer in the source tree. Future maintainers may not understand why the catalog is hand-curated.

## Risks / subtleties
- A reviewer doing a blind regex `s/sysinfo.resolve_device/session.resolve_device/` will get code that imports successfully and type-checks, but call sites that pass `resolve_device("auto")` will be interpreted as `resolve_device(ep="auto")` — hitting `expand_ep_name("auto")` and misbehaving. Such sites must be routed to `session.auto_detect_device()` instead. Static analysers will not flag this.
- Loss of the FIXME comment about ORT API limitations is a documentation regression — re-stating it on the `EPDeviceSpec` catalog would preserve institutional memory.
- The cache on `_get_available_eps` (`lru_cache(maxsize=1)`) was important because EP discovery touches DLL load paths and can be slow. The new `available_eps()` in `ep_registry` keeps the cache, but if any test or runtime path re-binds `WinMLEPRegistry` (the commit comment mentions tests do via `patch("winml.modelkit.session.ep_device.WinMLEPRegistry")`), the stale cache could mask the patch. This was implicit in the old code too but is now in a different module — risk surface unchanged, location changed.
- The transitive removal of the `"/"`-encoded multi-device value for OpenVINO is a *behavior* change, not a pure refactor. Any caller that filtered EPs by `if "/" not in device` is silently incorrect now.
